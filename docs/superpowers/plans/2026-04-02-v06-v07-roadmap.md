# Memento v0.6 – v0.7 Roadmap

**Goal:** 从 v0.5.1 到 v0.7.0 的优先路径。核心战略：先提升摄取质量（感知增强），再堵住摄取漏损（自动采集兜底），最后让后台处理更聪明（LLM Epoch 结构化）。

**Current Status:** v0.6.0 ✅ / v0.6.1 ✅ (2026-04-02)

---

## 战略排序依据

当前系统最大的瓶颈是 **ingestion coverage**，而不是 structuring sophistication：

1. `session_end` 只写 `sessions.summary`，不进 `capture_log` 也不进 `engrams`——高价值的会话总结完全脱离了 L2/L3 流转通道。
2. 如果 Agent 忘记调用 `memento_capture`，Epoch 再聪明也无米可炊。
3. 因此应先解决"没有被记住"的问题，再解决"如何更聪明地整理已记住的内容"。

此排序经架构评审、Claude Code 记忆系统启发文档（`借鉴 Claude Code 记忆系统的启发与行动项.md`）、及多轮独立 review 交叉确认。

## v0.6.0 — Agent 感知增强 (Agent Perception) ✅

*已完成（commit `65655dc`）。提升 Agent 读写判断质量的轻量改动。*

- [x] **staleness_level 字段** — `awake_recall` 返回 `fresh`/`stale`/`very_stale`，基于 FSRS v6 decay + rigidity 模型的启发式阈值（>0.6 / 0.3–0.6 / ≤0.3），MCP `memento_recall` 透传。
- [x] **Capture Exclusion Rules** — `memento_capture` 工具描述增加"不要记录"清单（代码结构、Git 历史、CLAUDE.md 已有内容、临时调试步骤、当前会话临时状态）。
- [x] **memento_prime 增强** — priming prompt 展示中性 staleness 标记（`⚠️较旧` / `⏳可能过时`）。
- [x] **awake_recall 检索升级** — 从纯 LIKE 升级为 vector → FTS5 → LIKE 三层 fallback，view_engrams 上使用 brute-force vector distance（正确但 O(N)，数据量上升后需评估专用向量索引）。
- [x] **decay.py 时区修复** — `_hours_since` helper 修复 aware/naive datetime 混用 TypeError。

**遗留验证项（可选）：**
- [ ] staleness 阈值边界测试补强（当前阈值是启发式初版，后续可按实际 recall 分布调整）
- [ ] FTS5 MATCH query sanitize（当前沿用 silent fallback，属于已知限制，非本版新引入风险）

## v0.6.1 — 摄取安全网 (Ingestion Safety Net) ✅

*已完成（commit `1e80839`）。堵住 session_end 数据流失漏洞。*

- [x] **Session-End 自动摘要兜底** — `session_end()` 在显式 captures + observations < 2 时，将 summary 通过 `awake_capture(origin='agent')` 补录为低信任 capture。直接查 `capture_log WHERE source_session_id = ?` 做抑制判断（awake 模式下 capture 不写 session_events）。
- [x] **保守去重/抑制逻辑** — content hash 去重（保守级，非语义去重），`origin='agent'`，strength 上限 0.5，不直接写 engrams。
- [x] **`memento://daily/today` 资源** — MCP Resource，合并当日 `capture_log` + `session_events`，按 `created_at` 排序。
- [x] **`memento_session_end` 描述更新** — 反映 fallback capture 行为。

**已知限制：**
- 去重为 content-hash 级，不是语义去重
- daily/today 使用 naive 日期前缀比较，非 timezone-aware

## v0.7.0 — LLM Epoch 结构化与再巩固 ✅

*代码已实现，测试已通过，审核缺陷已修复。后台处理从"记住"升级到"理解"。*

- [x] **LLM 基础设施** — `LLMClient` (llm.py) 支持 OpenAI 兼容 API、env vars 配置、指数退避重试、JSON mode。`epoch_run()` 已实现 LLM unavailable → auto-degrade to light 降级路径。
- [x] **L2 → L3 结构化** — Phase 2 接入 LLM：批量发送 capture_log（BATCH_SIZE=20），LLM 推导 type/tags，结构化后提升为 engrams。按 LLM 返回的 id 字段匹配（非数组下标）。content 变更时重算 content_hash + 清空 embedding + 设 embedding_pending=1。LLM 失败时 defer to cognitive debt。无 LLM 时保持 v0.5 auto-promote 兼容。
- [x] **Debt 回收闭环** — Phase 2 主动 JOIN cognitive_debt 捞回历史失败的 pending_consolidation 条目，与新数据去重后统一处理。
- [x] **再巩固 (Reconsolidation)** — Phase 5 接入 LLM：对 `can_modify_content(rigidity)=True` 的 engrams，基于 `recon_buffer.query_context` 调用 LLM 精炼内容。高 rigidity engrams 跳过。content 变更时同步更新 content_hash + 清空 embedding。LLM 失败时保留原内容。
- [x] **Prompt 模板** — `prompts.py` 提供 `build_structuring_prompt()` 和 `build_reconsolidation_prompt()`，纯函数，返回 JSON-mode prompt。
- [ ] **抽象化 (Abstraction)** — 聚类低频 engrams 形成抽象语义节点。推迟到 v0.8+。

**审核修复记录（commits `564f1a9` + `de3fd6b`）：**
- id 映射：从数组下标改为 LLM 返回的 id 字段，防止张冠李戴
- embedding 失效：Phase 2/5 content 变更时同步更新 hash + 清空 embedding
- 纯函数：Phase 2 用 item.copy() 避免原地修改脏引用
- merge_with：MVP 移除，避免无校验连环丢弃
- 批处理：BATCH_SIZE=20 防止 LLM context window 溢出
- Debt 回收：Phase 2 主动捞回历史失败条目

## 中期技术债务 — 图谱健康

*不阻塞 v0.6.x / v0.7.0 主线，但属于迟早要还的架构债务。*

- [ ] **Nexus 边衰减**
  - **What:** 对长时间未被 co-activate 的 nexus 边逐步降低 `association_strength`。
  - **Why:** 当前系统只对节点（engrams）做衰减，不对边做衰减。长期运行后图谱会变成致密毛线球，影响图游走的准度。
  - **Current schema:** `nexus` 表已有 `association_strength` 和 `last_coactivated_at` 字段，机制可以基于现有结构实现。
  - **Target:** v0.7.1 或 v0.8
